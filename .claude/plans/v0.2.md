# cowork-mdm v0.2 Plan

## 1. Strategic positioning

cowork-mdm is the IT admin's CLI for deploying Claude Desktop (aka Cowork) in the enterprise. It fills the gap between Anthropic's public documentation (8 of 51 MDM keys) and the app's actual configuration surface. Target user: Mac/Windows IT admin at a company deploying Claude Desktop with 3rd-party inference (Bedrock / Vertex / Gateway / Foundry).

v0.1 goal: profile generation + schema reference — MDM admin can produce a correct `.mobileconfig` or `.reg` in one command without hand-crafting XML.

v0.2 goal: add plugin marketplace management (macOS) + per-host doctor + TUI wizard. First release where a sysadmin can one-shot fully deploy Claude Desktop on a fleet.

## 2. Core decisions

### 2.1 Language: Go

- MacAdmins tooling precedent (Jamf CLI, Kandji CLI, Fleet, osquery all Go)
- Single static binary per-platform, zero runtime deps
- Trivial cross-compile: `GOOS=darwin GOARCH=arm64 go build` → one file
- First-class Windows support via `golang.org/x/sys/windows/registry`
- Go 1.23 minimum (stable, broad Homebrew availability)

### 2.2 Schema as source of truth

The 51-key schema is extracted **once** from `Claude.app/Contents/Resources/app.asar` (specifically the `FJ = me({...})` zod schema) and checked into `internal/schema/schema.json`. Consuming commands read this JSON via `//go:embed`.

Maintenance flow when Anthropic ships a new Claude Desktop version:
1. Maintainer runs `cowork-mdm schema extract --from /Applications/Claude.app`
2. Tool uses `@electron/asar` via `exec.Command` (or pure-Go unpacker) → parses `index.js` for FJ schema block → emits new `schema.json`
3. Maintainer commits + releases new cowork-mdm version

Users never extract; they consume the embedded JSON. Version skew is handled by the `extractedFromAppVersion` field in the JSON and a warning in `doctor` when the installed Claude.app drifts far from what schema.json was extracted from.

### 2.3 Profile abstraction

A `Profile` is an in-memory Go struct with typed per-key values. All output formats (mobileconfig, plist, jamf, intune, reg) are pure functions `func Export(Profile) ([]byte, error)`. One shared type, N encoders, zero duplication.

Input formats: parse `.mobileconfig` / `.plist` / `.reg` back into a `Profile`. Enables `profile edit` (read → mutate → write) and `profile diff`.

Templates live in `internal/profile/templates/*.yaml`, referenced by `--template bedrock-mcp` etc. YAML over JSON because templates are human-edited.

### 2.4 Plugin management layered on `org-plugins/` hardcoded path

Claude Desktop's `NF()` function hardcodes `/Library/Application Support/Claude/org-plugins` (macOS) and `C:\Program Files\Claude\org-plugins` (Windows) — not overridable. cowork-mdm manages this directory via:

- `marketplace add <git-url>` → `go-git` clone into `org-plugins/<repo-basename>/`, detect `plugins/` + `external_plugins/` subdirectories, symlink into top-level `org-plugins/`.
- `marketplace update` → pull all cloned marketplaces, rebuild top-level symlinks (handle new/removed plugins upstream).
- `plugin link` / `unlink` / `prune` → manage top-level symlinks without touching source repos.

v0.2 limits marketplace/plugin management to macOS. Windows `org-plugins/` requires junctions (`mklink /J`) + admin + Developer Mode — full support deferred to v0.3.

### 2.5 Doctor

`doctor` enumerates ~15 checks, each returns `(Status, Message, FixFn?)`. `--fix` invokes `FixFn` when present and status is not OK.

Checks (macOS):
1. Claude.app installed + version
2. `/Library/Managed Preferences/com.anthropic.claudefordesktop.plist` exists + parseable
3. MDM plist agrees with schema (no unknown keys, types match)
4. `/Library/Application Support/Claude/org-plugins/` exists + enumerable
5. Top-level entries are either directories (marketplaces) or symlinks (plugins) — no orphans
6. Each top-level symlink resolves (no dangling)
7. Each marketplace subdir has `.git` and is on a clean state
8. Per-user Claude-3p sessions, enabled plugins consistency
9. AWS credentials accessible if `inferenceProvider=bedrock`
10. Marketplace repos operable (go-git can read each cloned repo's HEAD)

Checks (Windows):
1. Claude.exe installed + version
2. Registry `HKLM:\SOFTWARE\Policies\Claude` parseable
3. Registry values agree with schema

`--json` emits machine-readable result for CI / MDM scripts.

### 2.6 TUI via bubbletea

`profile new` without `--template` triggers a bubbletea wizard: pick provider → fill provider-specific fields → pick MCP template → review → write file. Keeps non-interactive path (`--template x --out y.mobileconfig`) as first-class for scripting.

### 2.7 Testing strategy

- **Unit**: per-package `*_test.go` with table-driven tests. Schema validation, profile encoder roundtrips, path resolution.
- **Golden files**: `internal/profile/testdata/*.mobileconfig` / `*.reg` — generated samples compared byte-by-byte.
- **Integration**: `testing/integration/` — end-to-end CLI invocations using built binary, fixture inputs.
- **Platform-gated**: `//go:build darwin` / `//go:build windows` for OS-specific code; CI matrix runs both.
- **No live Claude.app dependency**: all tests use fixture `app.asar` under `testdata/`.

### 2.8 Release automation

- `.goreleaser.yaml`: `darwin/{amd64,arm64}` + `windows/{amd64,arm64}` + `linux/{amd64,arm64}` binaries, `.tar.gz` archives, Homebrew formula auto-push to `krislavten/homebrew-tap`, GitHub Release creation.
- No Apple Developer ID signing in v0.2 (accept Gatekeeper warning, note in README).
- CI: GitHub Actions — `go test ./...` on macOS+linux+windows, `golangci-lint run`, gated by `pull_request` and `push` to main.

## 4. Milestones + task breakdown

### M1 (foundation, serial)

- **T-01** `internal/schema/`: Go types + schema.json + loader + extract tool. Unblocks all downstream.
- **T-02** `internal/paths/`: platform-aware path resolver. Unblocks T-03..T-09.
- **T-03** `internal/profile/profile.go` + encode_mobileconfig.go: Profile struct, first encoder. Contract for encoders.

After T-01..T-03 merged, parallel work unlocks.

### M2 (parallel — profile encoders + parser)

- **T-04** `encode_plist.go` + `encode_jamf.go`: raw plist + Jamf Custom Settings text
- **T-05** `encode_intune.go`: Intune Preference File XML for macOS (see specs/profile.md — macOS-only for v0.2)
- **T-06** `encode_reg.go`: Windows `.reg` file
- **T-07** (uses T-02) `internal/marketplace/`: git clone + symlink lifecycle
- **T-08** (uses T-02) `internal/plugin/`: list/link/prune
- **T-09** (uses T-01, T-02, T-07, T-08) `internal/doctor/`: checks + --fix; reuses marketplace/plugin APIs for org-plugin checks
- **T-apply** (uses T-02, T-03) `internal/managed/`: platform-specific apply/status side effects

### M3 (parallel — UI + CLI wiring)

- **T-10** `internal/ui/`: bubbletea wizard
- **T-11** `cmd-wiring/`: cobra subcommand wiring — consumes T-03..T-10 and T-apply

### M4 (release)

- **T-CI** `.github/workflows/ci.yml`: test matrix + lint
- **T-RELEASE** `.goreleaser.yaml` + `.github/workflows/release.yml` + Homebrew tap template
- **T-DOCS** `docs/` polish + per-platform deployment cookbook

## 6. Release strategy

v0.2 is released **atomically**: no partial release. All T-01..T-11 + T-apply + T-CI + T-RELEASE + T-DOCS merged to main, `go build ./...` green, `go test ./...` green on macOS+Windows, CI green → tag `v0.2.0` → GoReleaser fires.

If any milestone M slips, we re-cut scope (maybe drop TUI, release as v0.2.0 without T-10). Coordinator decision.

## 7. Done definition

v0.2.0 ships when all checkboxes below are true:

- [ ] `go test ./...` green on macOS arm64, macOS amd64, windows amd64, linux amd64
- [ ] `golangci-lint run` clean
- [ ] `cowork-mdm schema list` prints ≥ 51 keys
- [ ] `cowork-mdm profile new --template bedrock-mcp --out /tmp/p.mobileconfig` generates a `plutil -lint`-valid mobileconfig
- [ ] `cowork-mdm profile export /tmp/p.mobileconfig --format reg` generates `.reg` parseable by `reg import` on Windows
- [ ] `cowork-mdm profile export --format jamf` output is pastable into Jamf Custom Settings UI
- [ ] `cowork-mdm marketplace add https://github.com/anthropics/claude-plugins-official` → clones + symlinks (tested in dry-run mode on CI)
- [ ] `cowork-mdm doctor` produces both human and `--json` output; returns exit 0 on a healthy host, non-zero otherwise
- [ ] Homebrew formula installs on fresh macOS
- [ ] `cowork-mdm --version` prints a non-dev version after release
- [ ] README documents every shipped command
- [ ] All 51 schema keys have a corresponding entry in `internal/schema/schema.json`

## 8. Out of scope for v0.2

- Windows `org-plugins/` management (junctions + admin) → v0.3
- Apple Developer ID signing → v0.3
- Automated schema re-extraction on new Claude.app releases → v0.3
- Bootstrap config server (`bootstrapUrl` endpoint helper) → v0.3
- Linux `systemd` integration beyond running the binary → v0.3+
- MCP server discovery / scaffolding → v0.3+

## 9. Risks + mitigations

- **Agents might write Go that compiles locally but fails cross-compile**: `verify.sh` runs `GOOS=windows go build` and `GOOS=darwin GOARCH=amd64 go build` in addition to native build.
- **schema.json format drift during v0.2**: frozen at M1 start. Any changes → coordinator chore PR + spec update.
- **Go dependency churn**: pin major versions in initial `go.mod`; no dep bumps during v0.2 except security.
- **Windows testing difficulty on a Mac**: rely on CI Windows runner; locally only verify cross-compile succeeds.
- **Claude.app version drift during v0.2**: pin `schema.json`'s `extractedFromAppVersion` to current (1.5354.0); doctor warns if installed ≠ pinned but tolerance ±2 minor.
